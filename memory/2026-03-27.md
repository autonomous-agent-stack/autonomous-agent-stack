## 原文要点全量摘记

- 这份“硬核版 v1 定义”剔除了前期容易导致项目失控的伪需求，把一个发散的想法收束成真正能在本地跑起来的工程级产品规格。
- 方案中几个非常精妙的工程取舍：
  - **硬件与生态的绝对绑定：** 锁死本地 Mac OS 和 M1 算力，利用 macOS Native STT 替换掉需要大量显存的本地 Whisper，并用 GLM-V API 解决视觉输入。
  - **权限与审批的梯度设计：** 不做“全量生物认证”，而是低危用 Bot 内联按钮确认，高危走 Mini App 结合生物特征和 `initData` 校验。Mini App 审批面板采用清爽浅色背景 UI，便于阅读终端日志、Git Patch 对比，降低高风险操作时的视觉疲劳和误读概率。
  - **身份与记忆的降维打击：** 一 Bot 双 Scope（`personal` / `shared`）彻底支持双人共用助理场景；底层元数据挂完整 actor 标签后，系统可精准识别当前交互者是你还是 Lisa；这样既能在专属私聊中保持记忆物理隔离，又能在共享群组中提供连贯协作上下文。
- 当前当务之急是停止做加法，冻结 v1 spec，进入实质性的代码拆解阶段。
- 五个实施包对应的里程碑蓝图：
  - **阶段一：稳定化与主路由收口 (Stabilization & Entrypoint)**  
    清理旧版 Webhook、遗留 Workflow 和僵尸代码；实现单一 App 主入口，统一挂载 Telegram、OpenClaw 兼容层、Panel 和 Admin 路由；修复 Spawn Gate 阻断问题，跑通安装和测试基线。
  - **阶段二：会话、身份与三层记忆模型 (Session, Identity & Memory)**  
    建立 `AssistantScope`、`ActorRole`、`CapabilityMode` 等核心 Enum/Type；实现 Telegram 白名单拦截器，拒绝非白名单私聊，群组强制鉴权并忽略环境噪音（仅响应 `@`、回复和指令）；基于 SQLite 落地 Session / Personal LT / Shared LT 的读写隔离与查询降级。
  - **阶段三：多模态媒体适配器接入 (Media Adapters)**  
    开发 `VisionAdapter` 接入 GLM-V API，处理 Telegram 图片下载、结构化解析并注入对话轮次；开发 `AudioTranscriptionAdapter` 桥接 macOS Apple Speech，实现大小/时长限制、异步转写及容灾降级的本地轻量 STT。
  - **阶段四：分级审批流与受控 Git 策略 (Approval & Git Policy)**  
    开发 Telegram Mini App 审批面板及高危动作验证逻辑，强制校验 `initData`、用户 ID、Action Hash 与时效；锁定 Git 执行策略，智能体只拥有创建分支和生成 Patch 的权限，主分支合并强制挂起等待人工核验。
  - **阶段五：安全技能库与受控多智能体并发 (Signed Skills & Bounded Agents)**  
    重构 `SkillRegistry`，关闭任意远程压缩包下载，强制签名白名单、固定版本号和能力声明校验；完善 Planner - Worker - Aggregator 并发架构，保证任务扇出时前端 Telegram 只保留单一、清晰的回复流。
- 结论：路线图依赖关系清晰，建议立即切入阶段一，先把单一入口架子和基础核心配置（Config Surface）接口规范用代码定义出来。
