## 14:30 - 三层架构最终确定

**完整架构图**:
```
客户入口(私人管家agent) 
    ↓
调度中心(control plane) 
    ↓
专业agent/worker
```

### 第1层: 客户入口(私人管家agent)
**定位**: 前台门面，不是后台内核
**技术选型**: 用OpenClaw现成做前台人格和聊天入口
**职责**:
- 聊天、收集需求
- 记住偏好
- 把自然语言变成标准任务单
- 汇报与确认
- 任务入口统一

**核心功能(4个)**:
1. 任务理解: "帮我把今天的订单录进系统" → 结构化任务
2. 偏好记忆: 常用系统/模板/审批习惯
3. 汇报与确认: 高风险前确认，完成后回报
4. 任务入口统一: 客户只面对一个"管家"

**不要做**:
- 长期人格宇宙
- 多agent互相辩论
- 自己改自己
- 完整CRM/ERP前台
- 再造一套skills生态

**实现方式**:
- 用OpenClaw现成能力
- 只加"任务翻译器"
- 让它把用户意图转成标准任务单
- 交给control plane

### 第2层: 调度中心(control plane)
**定位**: 系统秩序，不是首屏体验
**职责**:
- 派单
- 审批
- 日志
- 回放
- 权限
- 预算
- 治理和accountability

**入口**: 后台控制台(高级用户/管理员/实施同学用)
- 任务队列
- worker状态
- agent package
- 日志
- 审批
- 回放

### 第3层: 专业agent/worker
**定位**: 真正干活
**类型**:
- OpenClaw worker
- Linux worker
- Mac worker
- Win+影刀 worker

## 为什么这样分层

### 客户最容易理解
✅ "我跟一个管家说话，它帮我把事办了。"
❌ "这是一个异构执行控制平面，支持多worker、task gate、版本化agent package。"

第二句对你是架构，对客户是噪音。

### OpenClaw为什么适合当前台
- 官方定位: 个人、单用户、常驻助手
- 推荐路径: openclaw onboard(一步步配置gateway/workspace/channels/skills)
- 目标: 让用户尽快拥有可聊、可用、可常驻的personal assistant
- 公开支持: 多渠道(WhatsApp/Telegram/Discord/iMessage) + Web UI + onboarding + skills

### Control plane为什么适合当后台
- Paperclip等产品的公开定位: goal/task/budget/governance/accountability
- 它是"control plane"，不是聊天入口
- 最擅长: 系统秩序
- 不擅长: 首屏体验

## 决策记录

**为什么不在control plane里塞聊天窗口**
- Control plane擅长系统秩序，不擅长首屏体验
- 聊天是客户体验，应该在客户入口层
- OpenClaw天生就是做这个的

**为什么用OpenClaw而不是自建**
- OpenClaw公开就有完整的personal assistant能力
- 多渠道、Web UI、onboarding、skills都是现成的
- 重造这些太浪费时间
- 只需要加"任务翻译器"就够了

**为什么分三层而不是两层**
- 两层(控制平面→worker)缺少客户体验层
- 客户不容易理解"调度系统"这种架构词
- 三层(管家→调度→执行)才符合用户心智

**为什么私人管家不做人格宇宙**
- 18天内太重
- 先做"好上手的任务入口"
- 人格化以后可以慢慢加
