# 用 AAS + 管家 Bot + Claude Code CLI + ECC 落地“销售统计与提成发放”Excel 自动化的最佳实践深研报告

## 适用范围

这份文档回答 4 个问题：

1. 这类 Excel 提成业务为什么必须走“固定程序 + 可审计”。
2. 现在这套 AAS 仓库里，哪些基础已经有了，哪些还要补。
3. 原始 Excel 和最终交付 Excel 应该怎么定义成可执行合约。
4. 把歧义问清楚后，是否就能顺利把代码做出来。

结论先写在前面：

- 生产期不要让 AI 直接自由算钱。
- Claude Code CLI + ECC 应该放在“开发维护链”，不是“正式算账链”。
- AAS 应该负责入口、作业编排、审计、审批、回放。
- 最终是否能顺利出代码，取决于“歧义澄清 + 样本 + golden 输出 + 审计闭环”四件事是否齐全。

## 一句话结论

对于销售统计、提成核算、对账这类强规则业务，最佳实践是：

`管家 Bot 收件 -> AAS 创建确定性 job -> 固定程序处理 Excel -> 输出结果 + 异常 + 审计 -> 人工审核 -> 正式发放`

与此同时：

`Claude Code CLI + ECC -> 只负责开发/维护固定程序、规则文件、测试、文档`

不要把 Claude 当成运行期提成计算器。

## 为什么必须是“固定程序 + 可审计”

这类业务和通用问答不同，核心要求是：

- 同一份输入，任何时间重跑，结果必须一致。
- 每个数字都要能回溯到输入行、规则版本、程序版本。
- 发现异常时，必须知道是映射问题、规则问题、还是源数据问题。
- 最终发放前必须有人审，不能让 AI 在无人确认下改发放数据。

所以运行期应该收口为确定性链路：

1. 读取 Excel。
2. 标准化字段。
3. 去重、校验、归属。
4. 按版本化规则计算。
5. 生成异常报告。
6. 记录审计信息。
7. 进入人工审核。

AI 的位置应该是：

- 写解析器
- 写规则执行代码
- 补测试
- 修适配器
- 生成差异报告和文档

而不是在每次上传真实工资数据时临场推理。

## 当前仓库已经具备的基础

这条线不是从零开始。当前仓库里已经有 3 个关键基础：

### 1. Excel 相关的确定性引擎雏形

- `src/excel_audit/`
- `src/autoresearch/core/services/excel_audit.py`
- `src/autoresearch/api/routers/excel_audit.py`

当前更偏“核对/差异检查/对账”能力，不是完整的“多团队提成发放系统”，但方向是对的：运行期由固定程序执行。

### 2. 管家 Bot 到 Excel 任务的入口

- `src/autoresearch/core/services/butler_router.py`
- `src/autoresearch/api/routers/gateway_telegram.py`

目前 Telegram 管家入口已经能把“核对/提成/对账/excel/xlsx”等意图路由到 `excel_audit`，并异步执行后台任务。也就是说“Bot 触发 Excel 任务”的入口雏形已经存在。

### 3. AAS 的受控执行与审计思路

- `docs/agent-execution-protocol.md`
- `docs/task-brief-guide.md`
- `docs/decisions/deterministic-job-best-practice-v1.md`

这部分决定了你不需要把 Claude Code CLI 硬塞进生产期；它更适合做受控开发执行器，由 task brief 驱动，在隔离 workspace 里改代码、补测试、产出 patch。

## 推荐总体架构：两条链路分离

### 生产算账链

这条链只做真实业务处理：

`Bot/API 收件 -> AAS job -> 确定性 Excel 程序 -> 输出结果/异常/审计 -> 人工审核 -> 导出最终表`

要求：

- 不依赖 Claude 在运行时自由推理
- 必须可回放
- 必须可审计
- 必须有人工确认点

### 开发维护链

这条链只做程序变更：

`需求澄清 -> Task Brief -> Claude Code CLI + ECC -> 改代码/补测试/跑回归 -> 人审 -> 合并`

适合处理：

- 新增一种 Excel 模板
- 新增一套提成规则
- 修复字段映射问题
- 修复退款/补款逻辑
- 生成差异对比报告
- 更新 runbook 和业务说明

## 在 AAS 里的推荐落地方式

如果以当前仓库为基础继续做，建议分两层推进。

### 第一层：在现有 `excel_audit` 之上扩成“审计友好的 commission lane”

建议补这些模块：

- `src/autoresearch/api/routers/excel_ops.py`
- `src/autoresearch/core/services/excel_ops.py`
- `src/autoresearch/core/services/commission_engine.py`
- `src/autoresearch/core/repositories/excel_jobs.py`
- `src/autoresearch/shared/models.py`
- `tests/test_excel_ops_service.py`
- `tests/test_commission_engine.py`
- `tests/test_excel_ops_router.py`

职责边界建议：

- `excel_ops.py`
  - 负责收件、建 job、校验文件角色、触发 service
- `excel_ops.py service`
  - 负责完整处理链：读取、清洗、标准化、归属、计算、导出
- `commission_engine.py`
  - 只做确定性规则计算
- `excel_jobs.py`
  - 用 SQLite 记录审计信息和审批状态
- `tests/*`
  - 用固定样例和 golden 输出做回归

### 第二层：保留现有 Bot 路由，但改成双车道

Bot 层建议分成两类请求：

- 生产算账类
  - 调 `excel_ops` / `excel_audit` 这类固定程序 job
- 开发维护类
  - 调 AEP / Claude Code CLI 工作流，用来改程序，不直接给最终数字

这样“Bot 触发 Claude Code CLI + ECC”就不会污染正式算账链。

## Claude Code CLI + ECC 的最佳用法

### Claude Code CLI

它在这条线上的最佳位置是：

- 读取 task brief
- 读取 `CLAUDE.md` / docs / 真实样例说明
- 修改确定性程序
- 补测试
- 运行验证
- 产出 files changed / verification / manual follow-up

不要让它：

- 在正式发薪期直接读表后给最终金额
- 通过 prompt 临时决定某列是什么意思
- 用自然语言覆盖规则文件

### ECC

ECC 的价值不是“替你算钱”，而是把 Claude Code 的开发行为收口。

建议只 cherry-pick 使用：

- `/plan`
- `/tdd`
- `/code-review`
- `/verify`
- `/test-coverage`

以及少量 rules / skills：

- Python testing
- Verification loop
- Search-first
- Security / style

不建议把 ECC 全量塞进生产链，也不建议全量安装所有 skills 后再期待自动变稳。

## 原始 Excel：建议固定成 4 类输入角色

要让代码顺利落地，必须先把输入定义成角色文件，而不是模糊地说“几份业务表”。

### 1. 平台订单明细表

最小必需字段建议：

- 订单唯一标识
- 平台
- 下单/付款/完成日期之一
- 币种
- 原始订单金额
- 状态

常见风险：

- 订单号重复
- 金额是文本
- 状态不统一
- 多平台列名不一致

### 2. 销售跟单汇总表

最小必需字段建议：

- 销售人员标识
- 团队/岗位/区域/渠道
- 订单号或可关联键
- 归属比例

常见风险：

- 同一订单多人认领冲突
- 人员名不统一
- 比例和不等于 100%

### 3. 退款/调整表

最小必需字段建议：

- 关联订单号
- 调整类型
- 调整金额
- 发生日期或入账日期
- 原因

常见风险：

- 找不到关联订单
- 跨月冲减口径不清
- 重复调整

### 4. 提成规则表

最小必需字段建议：

- 规则版本号
- 生效日期
- 适用团队/岗位
- 计算口径
- 阶梯/比例/封顶/保底
- 特殊奖惩

常见风险：

- 文本规则有歧义
- 多条规则冲突
- 历史规则无法重放

## 最终交付 Excel：建议固定成 4 类输出

### 1. 标准化订单明细

用途：

- 作为唯一事实表
- 后续统计与审计的基础

建议元数据：

- job_id
- rule_version
- 输入文件摘要
- 程序版本
- 原始行引用

### 2. 异常清单

用途：

- 给业务/财务人工确认

建议元数据：

- 异常编码
- 严重级别
- 触发规则
- 原始文件/sheet/行号

### 3. 销售统计汇总表

用途：

- 给管理层看经营结果

建议元数据：

- 统计周期口径
- 汇总口径说明
- 规则版本

### 4. 提成发放表

用途：

- 给财务做最终发放依据

建议元数据：

- 规则版本
- 生效日期
- 舍入规则
- 审核状态

## AI 最需要先问清楚的歧义

把这些问题问清楚，是代码可落地的前提。

### 1. 周期口径

- 按下单日、付款日、完成日，还是回款日？
- 次月退款是回冲上月，还是记在本月？

### 2. 唯一键与去重

- 哪个字段是订单唯一键？
- 是否有子订单、拆单、合单？
- 多次导出如何去重？

### 3. 金额口径

- 提成按销售额、回款额、毛利，还是净额？
- 是否含税、含运费、含优惠？
- 部分退款如何冲减？

### 4. 销售归属

- 一笔订单是否允许多人分成？
- 主销售、协同、渠道经理、区域经理是否都参与？
- 比例从哪里来，能否人工覆盖？

### 5. 汇率与舍入

- 多币种按什么汇率折算？
- 汇率来源是什么？
- 保留几位小数？
- 四舍五入还是别的规则？

### 6. 规则版本边界

- 规则变更按生效日切还是按月份切？
- 月中改规则怎么处理？
- 历史重算时是否严格沿用旧版本？

### 7. 审核与权限

- 谁能上传原始表？
- 谁能忽略异常？
- 谁能确认生成正式发放表？

## 把歧义问清楚后，代码就一定能顺利出来吗

不能直接这么下结论。

“歧义澄清完”只是必要条件，不是充分条件。要让代码真正顺利出来，还需要 3 类资产：

### 1. 真实样本

至少准备 1 到 3 份真实 Excel 样本，并明确：

- 哪些列应该映射到什么内部字段
- 哪些行是异常
- 哪些输出是期望结果

### 2. Golden 输出

要有固定样例对应的期望输出：

- 标准化明细
- 异常清单
- 销售汇总
- 提成发放表

否则改一次规则就没有回归基线。

### 3. 审计/审批闭环

至少要记录：

- job_id
- 输入文件 hash
- 规则版本
- 程序版本
- 输出文件 hash
- 审核人和审核时间
- 异常处理动作

没有这部分，即使程序能跑，也不算“能上线”。

## 建议的实施顺序

### 第 1 阶段：先把强规则主链做稳

目标：

- 单团队
- 单套规则
- 单一日期口径
- 固定输出 4 份表
- SQLite 审计
- 人工审核后发放

### 第 2 阶段：再扩规则和异常能力

目标：

- 多团队
- 多规则版本
- 退款/补款/跨月调整
- 更强的异常分类
- 新旧规则差异报告

### 第 3 阶段：最后再加 AI 辅助能力

目标：

- AI 解释异常
- AI 帮忙适配新模板
- AI 生成管理摘要
- AI 生成规则变更对比说明

但注意：这些都应该加在确定性主链之上，而不是替代主链。

## 最终验收标准

如果要定义“现在可以开始让 Claude Code + ECC 出代码”的门槛，建议是这 4 条同时满足：

1. 歧义问题全部决策完毕。
2. 已提供 1 到 3 份真实样本。
3. 已提供 golden 输出或人工确认的期望结果。
4. 已确定最小审计/审批闭环。

满足这 4 条后，代码就不再是“看起来合理”，而更有机会一次做成“能验证、能维护、能上线”的工程结果。

## 推荐下一步

如果按当前仓库继续推进，最小可执行下一步是：

1. 先冻结输入/输出 Excel 合约。
2. 把上面的 7 类歧义整理成 intake checklist。
3. 在现有 `excel_audit` 之上补 `commission_engine + excel_jobs + excel_ops`。
4. 用 1 组真实样本和 1 组 golden 输出先跑通单团队试点。
5. 再把 Claude Code CLI + ECC 接到“开发维护链”，而不是正式算账链。

这才是用 AAS 落地这类 Excel 业务的稳妥做法。
